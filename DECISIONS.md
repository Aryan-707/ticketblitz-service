**CAP Theorem Tradeoff:** This system is not strictly CP; it is a pragmatic hybrid that degrades Availability at the Redis caching boundary (tripping open on failure to bypass degraded speeds) while exclusively shifting total Consistency burdens down to the absolute PostgreSQL unique constraints natively acting as the final unyielding authority.

## Individual Seat Tracking over Quantity Counters
I chose to track inventory via specific `Seat` entities with exact statuses instead of simple quantity counters. I considered just keeping the original integer decrements and adding pessimistic locks on the Tier. I made this choice because simple quantity counters cannot inherently protect against two users implicitly booking the same physically situated asset if the business model decides seats are assigned. It also makes oversold bugs physically impossible since 1 unique row = 1 owner. If I'm wrong, the database grows significantly larger row-by-row for 50,000-seat stadiums compared to a single integer row.

## Strict CP over AP for Distributed Lock Failure
I chose to throw a 503 Service Unavailable if the Redis `SETNX` lock operation throws an exception. I considered falling back to raw Database-level checking and locking. I made this choice because during flash sales, if Redis dies under extreme contention, pushing 100,000 requests directly to the DB without a distributed filter will crash the database instantly, paralyzing the entire platform. If I'm wrong, we lose revenue from simple connectivity blips because the entire checkout pipeline freezes when Redis restarts.

## DB Unique Index for Final Consistency
I chose to introduce a partial unique index on the database `UNIQUE(seat_id) WHERE status = 'CONFIRMED'`. I considered relying entirely on the application code logic plus the Redis lock. I made this choice because Redis can encounter split-brain scenarios, clocks could drift, or TTLs could expire precisely in the middle of a slow DB commit. The database MUST be the ultimate safety net. If I'm wrong, we just added a marginally heavier insert cost when pushing massive concurrent finalizations to the Orders table.

## Scheduled TTL Sweep over Redis KeySpace Notifications
I chose to use a Spring `@Scheduled` job every 2 minutes pulling `RESERVED` tickets older than 10 minutes to clean them up. I considered using Redis Expire hooks (Keyspace Notifications). I made this choice because Redis Keyspace events are fire-and-forget; if the backend is restarting exactly when a key expires, that notification is permanently lost and the seat is orphaned forever. A polling fallback is mandatory for absolute correctness. If I'm wrong, we induce continuous table scanning overhead on `Seat` every 120 seconds.
